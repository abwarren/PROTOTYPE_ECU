This document provides a subsystem-by-subsystem review covering 30 domains, scores each, and provides concrete, actionable recommendations with migration paths.

**Overall Engineering Score: 44/100**

## 4. Cross-Cutting Themes

### 4.1 The Architecture Is Correct But Untested

Every architecture document in `01_Architecture/` describes a plausible, well-reasoned system. The firmware architecture (3-layer stack, FreeRTOS, task priorities), the cloud architecture (MQTT → Kafka → TimescaleDB → REST API), the security architecture (HSM, secure boot, mTLS, OAuth 2.0) — these are all credible designs that a real automotive company would build.

However, none of these designs have been tested against reality. There is no prototype hardware to validate the pin assignments. No communication protocol has exchanged a single byte. No database has stored a single telemetry record. The designs are hypotheses awaiting falsification.

**Recommendation:** Begin TB-001 (Live RPM Pipeline) as early as possible — even with a development board and a simulator — to validate the architecture against real constraints.

### 4.2 The GPL Boundary Is the Single Most Important Architectural Decision

The project forks rusEFI (GPL-3.0) as its firmware foundation. The architecture documents describe a "platform layer" (`firmware/platform/`) that extends the upstream — but the GPL boundary is undefined:

- Does platform code link against GPL code? (If yes, it's GPL.)
- Are there well-defined IPC boundaries? (If yes, proprietary code can exist on the other side.)
- What is the clean-room strategy for proprietary algorithms?

This is not a Phase 5 concern. Every line of code written between now and Phase 5 is affected by this decision. Writing platform code without a clear GPL strategy risks contaminating the entire codebase.

**Recommendation:** WRITE ADR-0008 defining the GPL/Proprietary interface boundary before any platform code is written. See `ARCHITECTURE_IMPROVEMENTS.md` for the proposed design.

### 4.3 Safety Architecture Is Absent

The project describes an ECU that controls fuel, ignition, and electronic throttle on a running engine. Failure modes include uncontrolled acceleration, engine destruction, and vehicle fire. Yet there is no safety architecture document, no fault analysis, no watchdog hierarchy, no fail-safe state specification, no ISO 26262 analysis, and no ASIL decomposition.

The firmware architecture document mentions FreeRTOS and task priorities but contains zero safety analysis. The hardware architecture mentions AEC-Q components but no functional safety architecture.

**Recommendation:** WRITE `01_Architecture/safety-architecture.md` as a P0 specification. This must define fault classes, detection mechanisms, fail-safe states, watchdog hierarchy, and verification strategy. Even a prototype ECU controlling a real engine must have basic safety guards.

### 4.4 The Process Overhead Mismatches the Project Phase

At 18 commits and zero lines of custom code, the project enforces:
- A 20-agent specification program
- An 11-agent operational system (dormant)
- 16-point mandatory session checklist
- 12-document startup reading list
- Agent locking with cross-directory request system
- 6-level maturity model with transition ceremonies
- Branch-per-agent strategy with QA-only merge

This is appropriate for a 50-person team shipping production automotive software. At the current scale (one developer, research phase), it creates friction without corresponding benefit.

**Recommendation:** ADOPT process scaling per maturity level. See `PROCESS_IMPROVEMENTS.md`.

### 4.5 The Repository Has Excellent Structure But Critical Bugs

The 18-directory structure (ADR-0005) is well-organized and navigable. The DDD quality gate (33/33) is properly implemented. Session handoffs are comprehensive.

But three critical bugs remain unfixed from SESSION_001:
1. `.gitmodules` has a relative URL (breaks fresh clones)
2. CI triggers on `[main, develop]` but branch is `master` (CI never runs)
3. Submodule has 25K lines of generated build artifacts (masks real changes)

These are 15-minute fixes that have survived 5 commits since discovery.

**Recommendation:** Fix immediately. See `REPOSITORY_IMPROVEMENTS.md`.

### 4.6 The Database Is Referenced Everywhere, Designed Nowhere

Six tracer bullets reference a database. The cloud architecture describes TimescaleDB for telemetry. The workshop pipeline references customer and vehicle records. But no database schema exists — not even an entity-relationship diagram.

The domain model implied by the project is substantial:
- Vehicles (manufacturer, model, engine, ECU serial, firmware version)
- Calibrations (fuel/ignition tables, versioned, diffable)
- Customers (identity, assigned vehicles, tuning history)
- Telemetry (time-series RPM, temperatures, pressures, GPS)
- Workshops (multi-tenant, customer lookup, history)
- Firmware images (versioned, signed, A/B trackable)
- Users (RBAC: tuner, workshop owner, dealer, admin)

This is a complex multi-tenant system with time-series, versioned documents, and RBAC. Designing it after implementation begins will cause expensive migrations.

**Recommendation:** Design the database schema as a specification now. See `DATABASE_RECOMMENDATIONS.md`.

## 9. Final Assessment

The Prototype ECU project is at an inflection point. The process foundation is excellent. The architecture documents show domain expertise. The governance is ready.

What's missing is the first piece of working engineering — a single tracer bullet that proves the stack works.

**The strongest recommendation of this review: do not add more process. Do not add more documentation. Fix the critical bugs, complete the P0 specifications, then build TB-001. The project needs working code more than it needs another policy document.**
